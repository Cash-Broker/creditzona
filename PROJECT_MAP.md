## 1. Implemented Modules

### Lead Intake (Public)
- **Public lead form** at `POST /leads` with full validation (`StoreLeadRequest`), bot protection (honeypot + timing), Cyrillic-only name validation, phone normalization, 14-day duplicate phone prevention, and conditional guarantor/mortgage fields.
- **LeadService** handles creation inside a DB transaction: round-robin primary assignment (email-pool based), privacy consent PDF generation and storage, confirmation email dispatch, and Filament database notification to the assigned user.
- **LeadFormCompact.vue** — multi-step Vue form with amount slider (5k-50k), conditional guarantor/property sections, client-side Cyrillic validation, and success animation.
- **useLeadForm.js** composable owns all form state, validation rules, Latin-character stripping, and server error mapping.
- Rate limited: 5 submissions per 10 minutes per IP+UA.

### Lead CRM (Admin)
- **14 Filament resources** expose the Lead model from different perspectives: primary leads, attached leads, returned-to-me leads, returned lead archives, returned-to-me lead archives, and their respective edit/view/list pages.
- Lead lifecycle states: new, in_progress, sms, email, processed, approved, rejected, with a marked-for-later flag.
- **Assignment workflow**: primary assignment, additional (secondary) assignment, return-to-primary, archive attached, archive returned-to-primary. Each transition has dedicated service methods and Filament notification deduplication.
- **LeadForm** (shared schema, ~1000 lines): name splitting/composing, note history (NoteHistory support class), guarantor repeater with documents and rich-text notes, EGN encryption/masking, conditional field visibility by credit type and status.
- **LeadInfolist**: displays all lead data, guarantor documents with download links, privacy consent status and download, internal note history with custom Blade views.
- **LeadsTable**: 5-second polling, status badges with colors, marked-for-later checkbox, latest note preview tooltip, bulk mark/unmark actions, filters by assigned user and status.
- **LeadCommunicationWidget**: internal chat per lead using `LeadMessageService`.
- **LeadDailyHistoryWidget**: admin-only widget showing daily lead counts with date range filtering, Bulgarian locale.

### Contact Messages
- **Public contact form** at `POST /api/contact-messages` with validation (`StoreContactMessageRequest`), honeypot, Cyrillic-friendly fields.
- **ContactMessageService**: persists message, sends `ContactMessageReceived` mailable to configured recipient, supports admin assignment to operators, archiving, and lead generation from a contact message.
- **4 Filament resources**: ContactMessages (admin, unarchived), AttachedContactMessages (operator, assigned to me), ArchivedContactMessages (admin archive), AttachedContactMessageArchives (operator archive). Reusable `ContactMessagesTable` and `ContactMessageInfolist` schemas.
- Actions: assign (admin), archive (admin/operator), create lead from message.

### Blog
- **BlogResource** (Filament): full CRUD for blog posts with title, slug (auto-slugified), image upload, markdown content, excerpt, published status, scheduled publishing via `published_at`.
- **Public routes**: `GET /blog` (listing), `GET /blog/{slug}` (detail) — both server-rendered shell + Vue SPA.
- **API**: `GET /api/blogs` (published list), `GET /api/blogs/{slug}` (single post).
- **Vue pages**: `BlogPage.vue` (listing with BlogCard components), `BlogDetailsPage.vue` (full post with SEO).
- **useBlogStore.js** composable: 5-minute cache TTL, request deduplication, prefetch on hover, initial data seeding from `window.appConfig`.
- SEO: article schema, Open Graph tags, breadcrumbs generated in `PageController::blogShow()`.

### FAQ
- **FaqResource** (Filament): CRUD for questions with sort order and published toggle.
- **Public route**: `GET /faq` — server injects published FAQs as initial data; Vue renders `FaqPage.vue` with `FaqAccordion.vue`.
- **API**: `GET /api/faqs` returns published FAQs (question + answer only).
- SEO: FAQ schema markup generated in `PageController::faq()`.

### SEO System
- **Centralized config**: `config/seo.php` defines site metadata, business info, opening hours, social links, and per-page SEO (title, description, keywords, h1, robots, breadcrumbs, sitemap flag) for 9 pages.
- **SeoManager**: generates `SeoData` DTOs for backend, `browserPayload()` for frontend. Builds Organization, WebSite, Article, FAQ, and Breadcrumb schema.org markup.
- **Vue SEO** (`resources/js/seo/index.js`): `applyRouteSeo()` updates document title, meta tags, canonical, OG/Twitter tags, and structured data on every route change. Data-driven from `window.appConfig.seo`.
- **Sitemap**: `php artisan seo:generate-sitemap` generates `public/sitemap.xml` from configured routes + published blog posts.

### Contract Generation
- **ContractGenerationService** (~1130 lines): generates legal contract documents (PDF + DOCX) with ZIP archive. 9 document types (application request, mediation agreement/protocol, consultation agreement/protocol, company promissory note, loan agreement with 2 copies, co-applicant promissory note, declaration).
- **Supporting services**: `BulgarianDateFormatterService`, `BulgarianNumberToWordsService`, `CurrencyFormatterService` (EUR/BGN fixed rate 1.95583), `WorkingDayService` (weekend-aware date math).
- **ContractBatch model**: stores encrypted input payload, generated document paths, archive path. Auto-deletes stored files on model deletion.
- **ContractBatchResource** (Filament): create/edit forms with lead prefilling, company selection, conditional document type fields, client/co-applicant data, financial fields. View page with document download links and archive download.
- **File controllers**: `ContractBatchFileController` serves archive and individual document downloads from the `legal` storage disk.

### Document Management
- **AdminDocumentResource** (Filament): file upload/management with title, description, MIME-type whitelist (PDF, Word, Excel, images), 50MB limit, metadata sync.
- **AdminDocumentFileController**: inline open (PDF/images) and download with security headers (no-cache, no-sniff).
- **Lead document downloads**: 4 invokable controllers for lead documents, lead privacy consent, guarantor documents, and guarantor privacy consent PDF (dynamically generated).

### Calendar
- **CalendarEventResource** (Filament): custom manage page (not standard CRUD) integrated with FullCalendar.io. Event types: appointment, call, follow_up, task. Statuses: scheduled, completed, cancelled. Color-coded by type.
- **CalendarEventService**: CRUD with reminder state management, all-day event normalization, FullCalendar JSON payload generation, role-based ownership.
- **CalendarReminderService**: dispatches due reminders as Filament database notifications, 24-hour notification retention, automatic cleanup, reminder state reset on event update.
- **AdminCalendarEventFeedController** and **AdminCalendarEventTimingController**: JSON feed and drag-drop timing update endpoints.

### Admin Overview
- **AdminOverview widget**: stats dashboard with 5-second polling. Admins see: daily leads, new leads, contact messages, published blogs/FAQs. Operators see: my leads (total/new), returned-to-me leads.

### Authentication & Authorization
- **7 policies**: LeadPolicy (complex assignment-based access), ContactMessagePolicy, BlogPolicy (admin-only), FaqPolicy (admin-only), AdminDocumentPolicy (staff), ContractBatchPolicy (staff), CalendarEventPolicy (staff + ownership).
- **User model**: role-based (admin/operator) with hardcoded email pools for lead assignment eligibility. `canAccessPanel()` restricts Filament to admin/operator roles. `is_available_for_lead_assignment` toggle for operators.
- **Rate limiting**: 5 per 10 min for lead submissions and contact messages, keyed by SHA1(route+IP+UA).

### Privacy & Consent
- **Privacy consent PDF**: generated per lead via `LeadPrivacyConsentPdfService` using mPDF, stored on local disk at `lead-consents/Y/m/{ulid}.pdf`.
- **Public consent document**: static PDF served at a known path, metadata injected into every page via `renderPage()`.
- **EGN encryption**: `egn` field encrypted at rest on Lead and LeadGuarantor models. Masked in Filament UI (last 4 digits shown).

### Cookie Consent
- **CookieConsentManager.vue**: bottom banner + modal with granular preferences (necessary/analytics). Focus trapping, keyboard accessibility.
- **useCookieConsent.js**: localStorage persistence, policy version tracking, custom event dispatch.

## 4. Known Weaknesses

### Authorization Gaps
- Filament panel access is gated by `canAccessPanel()` (admin/operator), but there are **no per-resource `canAccess()` overrides** on most resources. Access control relies on policy checks and query scoping, which is functional but not defense-in-depth.
- Lead assignment pools are **hardcoded email arrays** in the User model (`LEAD_PRIMARY_ASSIGNMENT_EMAILS`, `LEAD_ADDITIONAL_ASSIGNMENT_EMAILS`). Adding new staff requires a code change and deployment.

### Missing Tests
- **Contact message email dispatch**: `ContactMessageSubmissionTest` has 3 tests but does not assert email content/recipient (only submission + validation).
- **Blog/FAQ API response shapes**: `FaqApiTest` has 1 test. No dedicated `BlogApiTest` exists (blog API is tested indirectly through page tests).
- **Filament access rules**: `AdminAuthorizationTest` covers lead-related policies well, but blog/FAQ/contract Filament resource access testing is minimal.
- **Contract generation edge cases**: `ContractGenerationServiceTest` has 4 tests covering happy path but does not test validation failures, missing fields, or document format errors.
- **Calendar feature tests**: `CalendarEventFeatureTest` has only 3 tests. No tests for event creation, update via UI, or deletion.
- **Cookie consent**: No tests for the Vue cookie consent system.
- **Rich content / NoteHistory**: Limited testing of note history edge cases (concurrent edits, malformed JSON).
- **Guarantor lifecycle**: No dedicated tests for guarantor document uploads, status changes, or privacy consent downloads beyond basic encryption checks.

### Code Quality Issues
- **No debug logging found in production code** (previously flagged `useContactForm` debug logs appear to have been removed).
- **Hardcoded Bulgarian strings** throughout Vue components and composables — no i18n system. Acceptable for a Bulgarian-only site but limits future localization.
- **LeadFormCompact.vue** is ~500 lines — a monolithic component that could benefit from splitting.
- **LeadForm.php** (Filament schema) is ~1000 lines — the largest single file in the Filament layer.
- **ContractGenerationService.php** is ~1130 lines — complex but well-structured with clear method separation.
- **No TypeScript** — all frontend JavaScript without type safety.
- **Mortgage credit type** appears commented out or limited in the public form (`LeadFormCompact.vue`), despite full backend support.

### Data / Schema
- `egn` on leads table is `nullable text` (not `encrypted` column type — encryption is handled by Eloquent cast, which is correct, but the column itself is plain text in the DB).
- `credit_bank` is a plain string field — mentioned in CLAUDE.md as needing encryption consideration for future sensitive data.
- Lead `status` is a plain string with no DB-level constraint — relies on application-level validation only.
- No database-level enum or check constraints on `credit_type`, `marital_status`, `property_type`.
